## 学習進捗

### ✅ 区分1: Webの基本概念 — World Wide Web とハイパーテキスト

#### 学んだこと
- Webの正式名称は **World Wide Web**（世界に広がるクモの巣）
- Web上の文書は **ハイパーテキスト** で構成されている
- ハイパーテキストの最大の特徴は、文書の中に別の文書への参照（**ハイパーリンク**）を埋め込めること
- `<a href="...">` タグがハイパーリンクそのもの

#### ハイパーテキスト vs プレーンテキスト
| 種類 | 特徴 |
|------|------|
| プレーンテキスト | 文字データのみ。他の文書への参照を埋め込む機能がない |
| ハイパーテキスト | 文書の中に別の文書への参照（リンク）を構造として埋め込める |

#### 重要な補足: ハイパーテキストとHTMLの関係
- **ハイパーテキスト** = 「文書間をリンクでつなげられるテキスト」という**概念・考え方**
- **HTML** = その概念を実現するために作られた**具体的な言語の1つ**
- HTMLの正式名称 **HyperText Markup Language** がそのまま関係を示している
- HTML以外のハイパーテキストシステムも歴史的に存在した:
  - **HyperCard**（1987年、Apple製）
  - **Gopher**（1991年、WWWの競合）
  - **Xanadu**（1960年代〜、ハイパーテキストという言葉の生みの親Ted Nelsonの構想）
- 現在も使われているもの: **Markdown**、**Wiki記法**（ただし最終的にHTMLに変換される）

#### HTMLが行っている2つの役割
1. **マークアップ**: コンテンツに構造・意味を与える（`<h1>`、`<section>`など）
2. **ハイパーリンク**: 他の文書・リソースへの参照を埋め込む（`<a>`、`<img>`など）

#### HTMLの階層構造について
- **HTML** = 構造を**記述する**役割（設計図）
- **ブラウザ（レンダリングエンジン）** = その記述を**解釈して画面に反映する**役割（職人）
- 「Webが認識する」のではなく、**ブラウザのレンダリングエンジンが解釈している**

### ✅ 区分2: Webを支える3つの発明 — URL・HTTP・HTML

#### 学んだこと
- ティム・バーナーズ＝リーが考案した3つの技術:

| 技術 | 役割 | 例え |
|------|------|------|
| **URL** | リソースの**場所**を示す（住所） | 送り先住所 |
| **HTTP** | リソースを**やりとりする手順**（通信ルール） | 「ください」→「はい、どうぞ」のやりとりの手順 |
| **HTML** | リソースの**中身を記述する**（文書の書き方） | 手紙の中身の書き方 |

#### HTTPの詳細理解
- HTTP = **HyperText Transfer Protocol**（ハイパーテキストを転送するための約束事）
- 基本は **リクエスト（要求）→ レスポンス（応答）** の一問一答
- プロトコル = 2者間の会話のルール（手順の決まり）

```
ブラウザ → サーバー：「/about.html をください」（リクエスト）
サーバー → ブラウザ：「200 OK。はい、これです」（レスポンス）
```

#### RESTについて
- RESTは「HTTPにある思想」ではなく、**HTTPの使い方を整理した設計思想**
- 2000年にロイ・フィールディングが提唱
- 核心: **「何を」（URL）と「どうする」（HTTPメソッド）を分けて設計しよう**

| メソッド | 意味 |
|---------|------|
| GET | 見せてください（取得） |
| POST | 新しく作ってください（作成） |
| PUT | これに差し替えてください（更新） |
| DELETE | 消してください（削除） |

#### RESTfulなAPI設計の例
```javascript
fetch('/articles', { method: 'GET' })       // 記事一覧を取得
fetch('/articles', { method: 'POST', body: 記事データ })  // 新しい記事を作成
fetch('/articles/3', { method: 'PUT', body: 更新データ })  // 記事3番を更新
fetch('/articles/3', { method: 'DELETE' })   // 記事3番を削除
```

#### fetch() の分解
| コードの部分 | 役割 |
|-------------|------|
| `fetch()` | サーバーにリクエストを送るJavaScriptの命令 |
| `'/articles'` | 送り先（URL） |
| `method: 'GET'` | 何をしたいか（HTTPメソッド） |
| `body: データ` | 一緒に送るデータ（作成・更新のときだけ必要） |

#### HTTPとRESTの関係の例え
```
HTTP = 「日本語が話せる」（通信できる能力）
REST = 「敬語のビジネスマナー」（その能力の使い方の流儀）
```

### ✅ 区分3: クライアントとサーバー

#### 学んだこと
- Webは **クライアント-サーバーモデル** で動いている

| 用語 | 何か | 具体例 |
|------|------|--------|
| クライアント | サービスを**要求する側** | Webブラウザ |
| サーバー | サービスを**提供する側** | Webサーバー |

- やりとりは**常にクライアントから始まる**（サーバーは自分からデータを送りつけない）

#### HTMLファイルの流れ
```
① HTMLをサーバーにアップロード → サーバー側にある
② ユーザーがアクセス → クライアントがリクエストを送る
③ サーバーがHTMLを返す → HTMLの「コピー」がクライアント側に届く
④ ブラウザが描画する → クライアント側で表示される
```
- 開発者ツールでHTMLを編集しても元サイトが変わらないのは、クライアント側のコピーを編集しているだけだから

#### ブラウザとブラウザエンジンの違い
- **ブラウザ** = アドレスバー、タブ、ブックマーク等を含む**アプリケーション全体**
- **ブラウザエンジン（レンダリングエンジン）** = ブラウザ内の**1つの部品**

```
ブラウザ（Chrome）
  ├── UI（アドレスバー、タブ、ブックマークなど）
  ├── ブラウザエンジン / レンダリングエンジン（Blink）← HTML/CSSを描画
  ├── JavaScriptエンジン（V8）← JavaScriptを実行
  ├── ネットワーク処理 ← HTTPリクエストを送受信
  └── その他（キャッシュ、セキュリティなど）
```

| ブラウザ | レンダリングエンジン |
|---------|-------------------|
| Chrome | Blink |
| Firefox | Gecko |
| Safari | WebKit |

#### ブラウザエンジンの仕事の範囲
- HTML/CSSを受け取る → 解析する → 構造データ（DOM/CSSOM）を作る → レイアウト計算 → 画面に描画
- JavaScript実行、通信、UI管理は**別の部品**が担当
- ただしJavaScriptエンジンとは連携する（DOM操作など）

## 重要な概念の整理

### Web / インターネット / HTML / ブラウザの関係

| 用語 | 何か |
|------|------|
| インターネット | 世界中のコンピューターをつなぐ**通信ネットワーク（道路網）** |
| Web | インターネット上で動く**サービスの1つ**（HTML + HTTP + URLの組み合わせ） |
| HTML | 構造を記述するための**言語（設計図）** |
| ブラウザ | HTMLを解釈して画面に表示する**ソフトウェア（職人）** |

### インターネット上のサービスはWebだけではない

| サービス | プロトコル | 何をするか |
|----------|-----------|-----------|
| Web | HTTP/HTTPS | Webページの閲覧 |
| メール | SMTP/IMAP | メールの送受信 |
| ファイル転送 | FTP | ファイルのアップロード・ダウンロード |
| DNS | DNS | ドメイン名→IPアドレスの変換 |

```
インターネット（道路網）
  ├── Web（宅配便サービス）
  ├── メール（郵便サービス）
  ├── FTP（引越し業者）
  └── その他いろいろ
```

### ✅ 区分4: 静的ページと動的ページ

#### 学んだこと

| | 静的ページ | 動的ページ |
|--|-----------|-----------|
| HTMLはいつ作られるか | **事前に**作って置いておく | **リクエスト時に**サーバーが生成する |
| 表示内容 | 誰が見ても**同じ** | ユーザーや状況によって**変わる** |
| 必要な技術 | HTML/CSS | HTML/CSS + サーバーサイド処理 |

- 動的ページの本質は「パーソナライズ」ではなく、**HTMLを事前に用意しているか、リクエスト時に生成しているか**
- WordPressのブログページは動的ページの例（PHPがDBからデータを取得してHTMLを生成）

#### テンプレーティングの概念
- どのCMS・言語でも共通する本質: **「テンプレート内にデータの挿入ポイントを置く」**
- 「雛形 + データ = 完成したHTML」という構造

| 要素 | WordPress | ヘッドレスCMS + TS |
|------|-----------|-------------------|
| テンプレート | PHPファイル | TSのテンプレートリテラルやJSX |
| 挿入ポイント | `<?php the_title(); ?>` | `${article.title}` |
| データの取得先 | 自分のデータベース | 外部APIへのfetch |
| データの取得方法 | PHPの関数が内部で処理 | fetch()でHTTPリクエスト |

#### レンダリング方式の分類

| 方式 | 概要 |
|------|------|
| 静的HTML | 手動で作ったHTMLをそのまま返す |
| SSG（Static Site Generation） | ビルド時にフレームワークがHTMLを自動生成。配信速度は静的HTMLと同等 |
| SSR（Server Side Rendering） | リクエストのたびにサーバーがHTMLを組み立てて返す |
| CSR（Client Side Rendering） | サーバーは空のHTMLとJSを返し、ブラウザ側でJSがHTMLを組み立てる |

#### レスポンス速度 vs ユーザーが画面を見られるまでの速度

| 観点 | 速い順 |
|------|--------|
| レスポンス速度（サーバーが返すまで） | 静的HTML ≒ SSG ≒ CSR > SSR |
| ユーザーが画面を見られるまで | 静的HTML ≒ SSG > SSR > CSR |

- CSRはサーバーの返事は速いが、届いた後にブラウザ側で処理が必要
- 速度と柔軟性はトレードオフの関係

#### 現代のフロントエンドの解決策: SSR/SSG + CSR の組み合わせ

```
① SSR/SSGで完成したHTMLを返す → ユーザーはすぐ画面が見える
② JavaScriptが読み込まれる
③ ハイドレーション: JSが各要素に振る舞いを結びつける
④ 以降はCSRとしてユーザーの操作に応じて画面を部分更新
```

- 「切り替わる」のではなく、**SSRが作った土台の上にCSRの機能が乗っかる**
- ハイドレーション = HTMLの見た目にJavaScriptの振る舞いを結びつける処理
- Next.js（React）やSvelteKitがこの仕組みを自動的に実現してくれる

#### 例え話
```
① 完成した家が届く（SSR） → すぐ住める。ただし電気が通っていない
② 電気工事が完了する（ハイドレーション） → スイッチと照明がつながる
③ 住人がスイッチを操作する（CSR） → 電気をつけたり消したりできる
```

#### CSRの使われ方
1. **ページ全体をCSR（SPA）**: ログイン後の管理画面など、SEO不要な場面
2. **部分的なCSR**: 初回はSSR/SSGで表示し、操作後はJSが部分更新（現代の主流）

#### 従来型MPA vs SSR の違い
- サーバー側の処理（HTMLを生成して返す）はほぼ同じ
- 違いは**ブラウザに届いた後**

| | 従来型MPA | SSR（Next.js/SvelteKit等） |
|--|----------|---------------------------|
| 初回表示 | サーバーがHTML生成 | サーバーがHTML生成（同じ） |
| 2回目以降の画面遷移 | 毎回サーバーにHTML丸ごと要求 | JSが部分的に画面を書き換える |
| ページ遷移時の体験 | 画面が白くなって再読み込み | スムーズに切り替わる |

#### SSG vs SSR の使い分け

```
SSG → 静的にHTMLを返す → ハイドレーション → CSR
SSR → 動的にHTMLを返す → ハイドレーション → CSR
                                          ↑
                              ユーザーが触るのはここ（どちらも同じ）
```

ゴール（CSR状態のページ）は同じ。そこに至るまでの入り口を要件に応じて選ぶ。

| | SSG | SSR |
|--|-----|-----|
| 表示速度 | ⚡ 最速 | 🔵 やや遅い |
| データの鮮度 | ❌ ビルド時点で固定 | ✅ 常に最新 |
| ユーザー別の表示 | ❌ できない | ✅ できる |
| サーバー負荷 | ⚡ ほぼゼロ | 🔴 高い |
| ビルド時間 | 🔴 ページ数に比例して増大 | ⚡ 関係なし |

判断基準:
```
そのページの内容は全ユーザー共通か？
  ├── YES → データは頻繁に変わるか？
  │     ├── NO  → SSG ✅
  │     └── YES → SSR or ISR
  └── NO  → SSR ✅
```

#### シニア視点からの訂正ポイント

**訂正1: 「SSGかSSRの2択」ではない**
- SvelteKitでは SSG / SSR / CSR only / Streaming SSR 等の組み合わせが可能
- `prerender`, `ssr`, `csr` を個別にon/offできる
- ソース: [SvelteKit公式 - Page options](https://svelte.dev/docs/kit/page-options)

**訂正2: 「ハイドレーション後は完全にCSR」は不正確**
- 同一ページ内の操作はCSR
- ページ遷移時はサーバー通信が発生する場合もある（`+page.server.ts`のload関数）
- ただし従来型MPAと違い、HTML丸ごとではなくデータだけ取得してJSが画面更新
- ソース: [SvelteKit公式 - Loading data](https://svelte.dev/docs/kit/load)

**訂正3: 「SSRは表示パフォーマンスを落とす」は条件付き**
- SSGで古いデータを表示→CSRで更新するとちらつき（CLS）が発生
- SSRは最初から最新データで表示するためちらつきがない
- CLSはGoogleのCore Web Vitalsでマイナス評価される
- ソース: [web.dev - CLS](https://web.dev/articles/cls)

#### ハイドレーションの詳細

**ハイドレーションで起きる2つのこと:**
1. **イベントの結びつけ**: ボタンクリック等の処理をHTMLの各要素に紐づける
2. **フレームワークがDOMの管理権を引き継ぐ**: 以降の画面更新をフレームワークが担当

**ハイドレーションのタイミング:**
- ユーザーのアクションとは無関係
- JSファイルのダウンロードと実行が完了した時点で自動的に行われる

```
0.0秒  ページにアクセス
0.3秒  HTMLが届いて画面が見える（SSR/SSGの仕事）
0.3秒〜 JSのダウンロードが裏で進行中（見えるが触れない状態）
0.8秒  ハイドレーション完了 → CSR状態になる（操作可能）
```

- この「見えるけど触れない」空白時間を**TTI（Time to Interactive）**と呼ぶ

#### フレームワーク（SvelteKit）はどこにいるのか

SvelteKitのコードはビルド時に**2つの成果物**に分かれる:

```
あなたが書いたSvelteKitのコード
  ↓ ビルド（npm run build）
  ├── サーバー用のコード（Node.jsで動く）
  └── クライアント用のJSファイル（ブラウザで動く）
```

| 場所 | SvelteKitの役割 | いつ動くか |
|------|----------------|-----------|
| サーバー側 | HTMLを生成して返す | リクエストを受けたとき |
| クライアント側 | DOMを引き継いで管理する | JSがダウンロード・実行されたとき |

**クライアント側のSvelteKitが届く仕組み:**
```html
<!-- サーバーが返すHTMLのイメージ -->
<!DOCTYPE html>
<html>
<body>
  <h1>記事タイトル</h1>
  <p>記事の本文...</p>
  <!-- ↑ ここまでがSSRで生成されたHTML -->

  <script src="/app.js"></script>
  <!-- ↑ この1行がクライアント側SvelteKitの読み込み指示 -->
</body>
</html>
```

サーバーが返したHTMLの中に `<script>` タグとしてJSファイルへの参照が含まれており、ブラウザがそれに従ってダウンロードする。このJSがクライアント側のSvelteKit。

### ✅ 区分5: Webの標準化

#### 学んだこと
- 標準化 = 「この技術はこういうルールで作りましょう」とみんなで共通の仕様を決めること
- 標準化がなければ、ブラウザごとに専用のHTMLを書いたり、出し分け処理が必要になる
- 実際に1990年代にはNetscape NavigatorとInternet Explorerが独自タグを作り、制作者が苦しんだ歴史がある

#### 主な標準化団体

| 団体 | 何を決めているか |
|------|----------------|
| **W3C**（ダブリュースリーシー） | CSS、アクセシビリティ等の仕様 |
| **WHATWG**（ワットダブリュージー） | HTML、DOM、Fetch等の仕様（Living Standard） |
| **ECMA International** | JavaScriptの仕様（ECMAScript） |
| **IETF** | HTTP、TCP/IPなどの通信プロトコル |

#### W3CとWHATWGの歴史的経緯
- 元々はW3CだけがHTML仕様を策定していた
- 2000年代初頭、W3CがHTMLの進化を止めてXHTMLに移行しようとした
- Apple・Mozilla・Operaのブラウザベンダーが反発し、2004年にWHATWGを設立
- WHATWGがHTML5の原型を作り成功 → W3CもXHTMLを諦めてHTML5を採用
- 2019年に合意: **HTMLの仕様策定はWHATWGが主導**することで決着
- 現在のHTMLの正式な仕様は「HTML Living Standard」（バージョン番号なし、常に更新）

#### XHTMLについて
- HTMLをXMLのルールで書き直したもの（文法が厳密）
- タグはすべて小文字必須、閉じタグ省略不可、属性値のクォート必須
- 最大の問題: 文法エラーが1つでもあるとページ全体が表示されない
- 現在Webサイト制作ではほぼ使われていない
- XMLの仕組み自体はRSS、SVG、sitemap.xml等で現在も活用されている

### ✅ 区分6: セマンティックWeb

#### 学んだこと
- セマンティック = 「意味の」「意味的な」
- セマンティックWebとは、Webページの情報に**機械が理解できる「意味」を持たせる**という考え方
- `<div>` ではなく `<header>`, `<nav>`, `<main>`, `<footer>`, `<article>` 等のセマンティックタグを使う

#### セマンティックHTMLのメリット

| メリット | 説明 |
|---------|------|
| SEO | 検索エンジンがページの構造を理解しやすくなり、適切に評価される |
| アクセシビリティ | スクリーンリーダーが要素の役割を読み上げられる |
| 保守性 | 開発者がコードを見たときに構造が理解しやすい |

- アクセシビリティが損なわれるとSEO的にも減点対象になる（GoogleのCore Web Vitals）
- セマンティックHTMLはクローラーにページの重要度の構造を伝える手段でもある

## Chapter 2: Webとネットワーク技術

### ✅ 区分7: コンピューターネットワークとは

#### 学んだこと
- コンピューターネットワーク = 複数のコンピューターを接続してデータをやりとりできるようにした仕組み

| 種類 | 正式名称 | 範囲 | 例 |
|------|---------|------|-----|
| **LAN** | Local Area Network | 狭い範囲 | 自宅のWi-Fi、オフィス内 |
| **WAN** | Wide Area Network | 広い範囲 | 離れた拠点同士をつなぐ |

- インターネットは世界中のLANやWANを相互接続した世界最大のネットワーク
- LANとWANは概念ではなく**物理的な設備**（ケーブル、ルーター等）で構成されている
- ただしLANとWANの境界には揺らぎがある（厳密な境界線はない）

#### 実用的な分類の考え方
```
LAN = 自分（や自社）が管理しているネットワーク
WAN = 自分の管理外のネットワーク（プロバイダや通信事業者が管理）
```

#### データがサーバーに届くまでの経路
```
あなたのPC
  ↓ Wi-Fi電波
Wi-Fiルーター（ここまでがLAN）
  ↓ 光ファイバーケーブル
プロバイダの設備（ここからWAN）
  ↓ 光ファイバー・海底ケーブル等
サーバーがあるデータセンター
```

### ✅ 区分8: TCP/IPとは

#### 学んだこと
- TCP/IP = インターネットでデータをやりとりするためのプロトコルの組み合わせ
- 役割ごとに**階層（レイヤー）に分けて**、それぞれ別のプロトコルが担当する設計

#### TCP/IPの4階層モデル

```
第4層: アプリケーション層     ← HTTP, SMTP, DNS, FTP など
第3層: トランスポート層       ← TCP, UDP
第2層: インターネット層       ← IP
第1層: ネットワークインターフェース層 ← Wi-Fi, イーサネットなど
```

#### 各層の役割（配送の例え）
```
第4層: 手紙の内容を書く（HTTPメッセージを作る）
第3層: 届けるための管理（データ分割・番号付け・到着確認）
第2層: 宛先を決めてルートを選ぶ（IPアドレスで届け先を決定）
第1層: 実際に運ぶ（電気信号・光信号・電波で物理的に送る）
```

#### 階層に分けるメリット
- **関心の分離（Separation of Concerns）**: 各層が自分の仕事だけに集中し、他の層の内部実装を知らなくていい
- 第1層をWi-FiからLANケーブルに変えても上の層は影響を受けない
- 第4層にHTTPでもSMTPでも使えて下の層は同じ仕組みが使える
- この考え方はフロントエンド開発のコンポーネント設計にも通じる

### ✅ 区分9: WebサーバーとHTTP

#### 学んだこと
- Webサーバー = Webブラウザからのリクエストに応じてコンテンツを返す常駐プログラム
- 代表的なWebサーバーソフトウェア: Apache、IIS、Nginx

#### HTTPの重要な特性: 環境非依存
- HTTPという共通プロトコルに従っている限り、OS・ブラウザ・サーバーソフトの違いに関係なく通信できる
- ただし**HTTPが保証するのは「同じデータが届く」ことまで**
- 「同じ見た目で表示される」かは別問題（レンダリングエンジン、デフォルトCSS、フォント等の差異がある）

| 段階 | 統一の仕組み | 保証される内容 |
|------|------------|-------------|
| 通信 | HTTP | 同じデータが届く |
| 表示 | W3Cの標準仕様 | 近い見た目になる（完全一致ではない） |

### ✅ 区分10: Webページが表示される流れ

#### 学んだこと
- Webページの表示は1回の通信で完結せず、**複数回のHTTP通信の積み重ね**

#### 表示までの流れ（通信の観点）
```
1. URLをブラウザに入力
2. DNSサーバーにドメイン→IPアドレスの問い合わせ（UDP）
3. 取得したIPアドレスに対してTCP接続確立（3ウェイハンドシェイク）
4. HTTPリクエストでHTMLを取得
5. ブラウザがHTMLをパースし、CSS・画像・JS等への参照を発見
6. それぞれ追加のHTTPリクエストで取得
7. 全リソースが揃った段階でレンダリング完了
```

#### HTTPはあらゆるファイル形式を転送可能
- HTML、CSS、JS、画像、動画、JSON、フォント等すべてHTTPで送受信される
- サーバーはContent-Typeヘッダーでデータの種類をブラウザに伝える

| ファイル | Content-Type |
|---------|-------------|
| HTML | `text/html` |
| CSS | `text/css` |
| JavaScript | `application/javascript` |
| JPEG画像 | `image/jpeg` |
| JSON | `application/json` |

#### UDPでのDNS問い合わせが失敗した場合のリトライ

| ケース | 判断の仕組み | タイミング | その後の動作 |
|--------|------------|-----------|------------|
| 応答なし | タイムアウト | 2〜5秒後 | 同じサーバーにリトライ → 別サーバーに切替 |
| データ破損 | UDPチェックサム | 受信した瞬間 | パケット破棄 → タイムアウト → リトライ |

- どちらのケースも最終的に同じリトライの流れに合流する
